# 징계 챗봇 AI Canvas 설계안 v2

작성일: 2026-03-05
대상: 외부 법률정보 + 내부 징계 DB 결합형 챗봇

## 1) 문서 목적
- 기존 Step1 워크플로우를 전면 폐기하지 않고, 운영 안정성과 답변 실용성을 높이는 v2 설계를 정의한다.
- 목표는 다음 2가지를 동시에 달성하는 것이다.
1. 일반 법률 질의는 검색 가능한 공식 근거를 기반으로 가능한 범위까지 답변
2. 내부 징계 DB를 결합해 회사 실무 맥락(유사사례/양정 포인트)까지 보강

## 2) 현재 문제와 설계 방향
현재 문제:
- 외부 API 일부 실패 시 전체 파이프라인이 중단됨
- 근거가 조금만 부족해도 차단응답으로 끝나 사용자 체감 품질이 낮음
- 소스별 컬럼 구조가 달라 후속 노드가 자주 깨짐

v2 방향:
1. 부분 성공 허용: 일부 소스 실패여도 답변은 생성
2. 근거 스키마 통일: 법령/판례/해석/내부DB를 같은 구조로 정규화
3. 하드 차단 최소화: 공식근거 0건 또는 금지 도메인일 때만 차단
4. 내부DB 보강: 법적 판단은 공식근거 우선, 내부 사례는 실무 보조

## 3) 답변 정책 (중요)
- 기본 정책: "검색 가능한 범위 내 우선 답변"
- 하드 차단 조건:
1. 공식근거 0건
2. 금지 도메인 포함
- 소프트 경고 조건:
1. 공식근거 1건
2. 필수 슬롯 일부 누락
- 소프트 경고 시 동작:
1. 답변 생성은 진행
2. `[불확실/부재]`에 누락 항목 명시
3. 다음 액션(추가 사실/기간/키워드) 제시

## 4) 아키텍처 v2

### 4-1. 계층 구조
1. 질의 정규화 계층
- 사용자 질문을 `쟁점/키워드/기간/질문의도`로 구조화

2. 증거 수집 계층
- 외부: 법령/판례/해석 API
- 내부: 징계 DB(API 또는 Dataset)

3. 증거 정규화/스코어 계층
- 모든 증거를 표준 Evidence 스키마로 변환
- 신뢰도/최신성/직접성 점수 부여

4. 답변 생성 계층
- 표준 스키마 기반 템플릿 생성
- 결론, 근거링크, 불확실/부재 필수 출력

5. 게이트/전달 계층
- 하드 차단 여부만 최종 판정
- 통과 시 답변 전달, 실패 시 차단 메시지 전달

### 4-2. 신뢰도 우선순위
1. 법령 본문
2. 판례 본문
3. 행정해석 본문
4. 내부 징계 DB(보조 근거)

## 5) 표준 Evidence 스키마 (통합 규격)
아래 컬럼으로 외부/내부 소스를 모두 변환한다.

| 컬럼 | 설명 |
|---|---|
| `evidence_id` | 고유 ID |
| `source_type` | `law` / `precedent` / `interpretation` / `internal_case` |
| `title` | 법령명/사건명/사례명 |
| `authority` | 법원/기관/사내 규정 출처 |
| `date` | 선고일/시행일/의결일 |
| `key_point` | 핵심 문장 요약 |
| `quote` | 원문 발췌(짧게) |
| `url` | 원문 링크(내부 DB는 내부 참조키) |
| `jurisdiction` | 국가법/사내규정 등 |
| `score` | 신뢰도 점수(0~1) |
| `is_official` | 공식 근거 여부(boolean) |
| `tags` | 쟁점 태그(괴롭힘, 징계양정 등) |

## 6) AI Canvas 노드 설계 v2 (권장)

### 6-1. 핵심 노드 흐름
1. `에이전트`
2. `에이전트 메시지 가로채기`
3. `에이전트 프롬프트_질의정규화`
4. `파이썬_정규화파싱`
5. `파이썬_URL생성`
6. `API_법령목록`
7. `API_판례목록`
8. `API_해석목록`
9. `파이썬_법령상세URL`
10. `파이썬_판례상세URL`
11. `파이썬_해석상세URL`
12. `API_법령본문`
13. `API_판례본문`
14. `API_해석본문`
15. `API_내부징계DB조회` 또는 `데이터셋_내부징계DB`
16. `파이썬_근거표준화` (외부+내부를 Evidence 스키마로 변환)
17. `데이터 연결_근거통합`
18. `파이썬_근거스코어링`
19. `에이전트 프롬프트_답변생성`
20. `파이썬_검열게이트`
21. `데이터 조건 분기`
22. `프롬프트_차단응답`
23. `에이전트로 전달`

### 6-2. 부분 성공 허용 규칙
- 6~14 중 일부 실패해도 16~23을 진행한다.
- 단, 공식근거(`is_official=true`)가 최종 0건이면 22로 분기한다.

### 6-3. 내부DB 결합 규칙
- 내부DB가 비어도 워크플로우는 계속 진행한다.
- 내부DB는 법적 결론의 단독 근거로 사용하지 않는다.
- 내부DB는 `유사사례`, `양정 비교`, `사내 절차`에만 사용한다.

## 7) 프롬프트 정책 v2

### 7-1. 답변생성 프롬프트 핵심 지시
1. 공식근거 1건 이상이면 답변 생성
2. 근거 부족 부분은 `부재`로 표시
3. 일반론(요건/입증 포인트)은 작성 가능
4. 단정 금지, 링크 없는 주장 금지

### 7-2. 차단 프롬프트 사용 조건
- `official_evidence_count == 0` 또는 `domain_not_allowed == true`일 때만 실행

## 8) API 안정화 전략

### 8-1. 단기 운영 (현재 구조 유지)
1. 워밍업 호출 1회
2. 목록 API 순차 실행 (`6 -> 7 -> 8`)
3. 실패 노드만 1~2회 재시도
4. 본문 API 실행 전 상세 URL 비어있지 않은지 확인

### 8-2. 중기 구조 개선 (권장)
- `law.go.kr` 앞단에 중계 API를 둔다.
- 중계 API가 재시도(예: 2회), 백오프, 타임아웃, 캐시를 담당한다.
- AI Canvas는 중계 API만 호출한다.

## 9) 내부 징계 DB 설계

### 9-1. 최소 스키마
| 컬럼 | 설명 |
|---|---|
| `case_id` | 사건 ID |
| `violation_type` | 위반유형 |
| `fact_summary` | 사실 요약 |
| `discipline_level` | 징계수위 |
| `aggravating` | 가중사유 |
| `mitigating` | 감경사유 |
| `rule_ref` | 사내규정 조항 |
| `decision_date` | 의결일 |
| `notes` | 비고 |

### 9-2. 보안 원칙
1. 개인식별정보 비식별화
2. 최소권한 API 키 사용
3. 로그에 PII 저장 금지

## 10) 모델/토큰 권장값
1. 질의정규화 노드: `gpt-5-mini`, max tokens `700`
2. 답변생성 노드: `gpt-5.2`(또는 `gpt-5`), max tokens `4500` (필요 시 `6000~7000`)
3. 차단응답 노드: `gpt-5-nano`, max tokens `250`

## 11) 단계별 전환 계획
1. Phase 1 (1~2일): 게이트 완화 + 부분 성공 허용
2. Phase 2 (2~4일): Evidence 표준화 노드(16)와 스코어링 노드(18) 도입
3. Phase 3 (3~5일): 내부DB 조회 노드(15) 연결
4. Phase 4 (운영): 중계 API 도입 및 실패율 모니터링

## 12) 완료 기준 (DoD)
1. 일반 법률 질의에서 공식근거 1건 이상이면 차단 없이 답변
2. 공식근거 0건인 경우에만 차단응답
3. 내부DB가 없어도 답변 생성, 있으면 유사사례 섹션 보강
4. 목록/API 일부 실패 상황에서도 파이프라인 전체 중단률 감소
5. 응답에 `[불확실/부재]`가 일관되게 출력

## 13) 즉시 적용 체크리스트
1. 노드20(검열게이트) 정책을 `evidence>=1 통과`로 반영
2. 노드21 조건분기 True/False 매핑 점검
3. 노드23 전달 컬럼을 `output_response*`로 고정
4. 내부DB 연결 전에도 문답 품질 테스트 10건 수행
5. 내부DB 연결 후 동일 테스트셋 재평가

---
이 문서는 Step1 자산을 유지하면서 운영 품질을 올리는 v2 기준 설계서다.
핵심은 "하드 차단 최소화 + 근거 기반 답변 최대화 + 내부DB 보강"이다.
