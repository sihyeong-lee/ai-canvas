# AI Canvas 기반 징계 챗봇 구축 계획서

## 0. 문서 목적
- 본 문서는 AI Canvas 플랫폼에서 "징계 판단 지원 챗봇"을 구현하기 위한 실제 구축 계획서다.
- 목표는 `공식 원문 근거(법령/판례)` + `내부 징계 DB`를 결합한 근거 강제형 응답 시스템이다.
- 단순 LLM 답변이 아니라, 근거가 없는 문장을 출력 단계에서 차단하는 구조를 기본 원칙으로 둔다.

## 1. 전제 및 요구사항 정리
- 목표 서비스: 회사 내부 징계 검토 지원 챗봇
- 핵심 요구:
- 법령/판례를 공식 채널에서 수집
- 내부 징계 사례/규정 DB를 함께 조회
- 답변 시 인용 슬롯 강제
- 근거 부족 시 "부재/불충분" 명시 강제
- 절차 질문 시 전자소송/실무 절차까지 안내 가능
- 사용자 기대 출력 톤:
- `[🟢 Online Mode | yyyy.MM.dd_HH:mm:ss]`
- `🔍 Search Strategy`
- `[🌐 웹 서치 적용: ...]`
- 법령/판례/전략/다음 액션 구조

## 2. AI Canvas 기능 숙지 결과 (18개 PDF 반영)
- 플랫폼 기본:
- 노코드 워크플로우, 에이전트 워크플로우, RAG 구성 가능
- 워크스페이스 협업, View/Edit 권한, 링크 공유
- 캔버스/노드/엣지 관리, 플로우 설정(이벤트 기반 실행), 스케줄 실행
- 챗봇 구현 핵심 노드:
- `UI > 에이전트`: 대화형 에이전트, 웹검색, 메시지 가로채기/전달 확장
- `UI > 챗 UI`: 모델 선택, PDF 입력, System Prompt 설정
- `데이터 > 에이전트 메시지 가로채기`, `데이터 > 에이전트로 전달`
- `API > 에이전트 프롬프트`: 툴 사용형 추론(웹검색, 계산 등)
- 외부 연동 핵심:
- `API > 커스텀 API`: URL/Method/Header/Body/JSON 변환 지원
- `API > API 데이터`: 워크플로우 API 입력 스키마 정의
- `배포 > 워크플로우 API 배포`: 외부 POST 호출 가능한 엔드포인트 생성
- `배포 > 데이터셋 API 배포`: 결과 데이터 API 제공, Swagger 테스트
- 데이터 처리 핵심:
- `데이터 > PDF 데이터`, `PDF-데이터 변환`, `URL 리더`, `데이터 저장소`
- `전처리 > 텍스트 전처리`, `텍스트 임베딩`, `파이썬 스크립트`
- `모델 > 유사도 군집화`(중복/유사 근거 정리), `시각화/리스트` UI 출력
- 운영/배포 핵심:
- 앱 배포 시 접근권한(누구나/로그인/초대)
- 플로우 이벤트로 메뉴 이동/노드 실행/탭 변경 가능
- 스케줄 등록으로 주기적 수집/동기화 가능

## 3. 목표 아키텍처 (근거 강제형)
- 레이어 1: 입력/의도 분석
- 사용자 질의 정규화(사건유형, 사실관계, 시점, 목적)
- 쟁점/요건사실 추출
- 레이어 2: 근거 수집 (공식 원문 + 내부 DB)
- 법령: 공식 Open API로 조문/시행일/개정이력 수집
- 판례: 공식 판례 검색/상세 채널 수집
- 절차: 전자소송/실무 절차 채널 보강
- 내부DB: 사규/취업규칙/징계양정표/과거 사례 검색
- 레이어 3: 근거 정합성 검사
- 조문번호 형식, 시행일, 사건번호 형식 자동 점검
- 사실관계-쟁점-결론 1:1 매칭 검증
- 레이어 4: 생성 및 출력 강제
- 템플릿 기반 생성(필수 슬롯 누락 시 반려)
- 근거 없는 문장 제거 후 재생성
- 부재 근거는 "부재/불충분"으로 강제 표기

## 4. 데이터 소스 정책
- 공식 원문 소스(우선순위):
- 1순위 법령: 국가법령정보 공동활용 Open API
- 1순위 판례: 국가법령정보센터 판례
- 2순위 판결문 보강: 사법정보공개포털
- 절차 보강: 전자소송 포털
- 공공데이터 메타: data.go.kr API 메타 참조
- 내부 소스:
- 인사규정/취업규칙/단체협약
- 징계양정표/징계위원회 운영 규정
- 과거 징계 사례 요약(비식별화)
- 감사/컴플라이언스 지침

## 5. AI Canvas 구현 설계 (노드 단위)

### 5.1 입력/대화 채널
- `UI > 챗 UI` 또는 `UI > 에이전트`를 메인 진입점으로 사용
- `System Prompt`에 필수 출력 슬롯 및 금지 규칙(무근거 서술 금지) 명시
- 질의 전문/메타(사용자ID, 시간, 채널) 저장용으로 `데이터 저장소` 연결

### 5.2 질의 정규화 파이프
- `데이터 > 텍스트 입력`으로 구조화 필드 입력도 지원
- `API > 에이전트 프롬프트`로 저온 추론:
- 사건유형(노동 징계)
- 사실관계
- 쟁점 리스트
- 필요 증거
- 출력 JSON 스키마 고정

### 5.3 외부 법령/판례 수집
- `API > 커스텀 API`를 통해 공식 엔드포인트 호출
- 요청 헤더/토큰 관리
- 응답 JSON 자동 변환
- 법령 조회 파이프:
- 법령 목록 조회 -> 본문/조문 상세 조회 -> 시행일 기준 필터
- 판례 조회 파이프:
- 키워드/법원/선고일 필터 -> 사건번호 기준 상세
- 보강 수집:
- 사법정보공개포털/전자소송 절차 안내 페이지는 `URL 리더` 또는 API 연계

### 5.4 내부 DB 조회
- 방식 A: 사내 DB를 사내 API로 제공 후 `커스텀 API` 호출
- 방식 B: 배치로 CSV/시트 적재 후 `데이터 저장소` 조회
- 내부DB 최소 스키마:
- `case_id`, `위반유형`, `행위요약`, `징계수위`, `감경/가중`, `재심여부`, `근거규정`, `비고`

### 5.5 근거 결합/RAG
- 외부 근거 + 내부 근거를 하나의 근거 테이블로 합침
- `전처리 > 텍스트 전처리`로 정규화
- `전처리 > 텍스트 임베딩` + `모델 > 유사도 군집화`로 중복 근거 정리
- 최종 컨텍스트는 "근거 chunk 묶음"만 LLM에 전달

### 5.6 출력 강제/검열
- `전처리 > 파이썬 스크립트` 또는 `UI > 샌드박스`로 규칙 검증
- 검증 규칙:
- 법령 슬롯(법령명/조문/시행일) 누락 시 실패
- 판례 슬롯(법원/선고일/사건번호) 누락 시 실패
- 근거 링크 없는 단정문 비율 임계치 초과 시 실패
- 실패 시 재생성 루프
- 성공 시 최종 응답 포맷으로 확정

### 5.7 배포
- 사용자용:
- `애플리케이션` + `배포` 노드로 웹 챗봇 배포
- 접근권한: 초대 사용자 또는 로그인 사용자 권장
- 외부 연동용:
- `API 데이터` + `워크플로우 API 배포`로 API 엔드포인트 제공
- 운영팀 대시보드:
- `페이지/리스트/차트`로 질의량, 근거 미충족률, 재생성률 시각화

## 6. 출력 템플릿 (강제 포맷)
- 아래 형식으로 시스템 규칙을 박아야 한다.

```text
[🟢 Online Mode | {timestamp}] >
🔍 Search Strategy >
> {수집 전략 요약}
> [🌐 웹 서치 적용: {도메인 목록}]

🧊 사실관계 요약
- {사용자 진술 요약}
- {불명확 포인트}

⚖️ 쟁점
- {쟁점1}
- {쟁점2}

📚 적용 법령
- 법령명: {name}
- 조문: {article}
- 시행일: {effective_date}
- 해석 포인트: {point}

🧑‍⚖️ 관련 판례
- 법원: {court}
- 선고일: {date}
- 사건번호: {case_no}
- 판시요지: {holding}
- 근거 링크: {url}

🏢 내부DB 참고
- 사내 규정: {rule_name}
- 유사 징계사례: {case_id / 요약}
- 차이점: {difference}

♟️ 종합 판단
- {종합 의견}

🚀 다음 액션
- {필요 증거}
- {문서/절차}
- {기한}

[불확실/부재]
- {근거 부족 항목 명시}
```

## 7. 신뢰성/안전성 설계
- 환각 방지:
- 모델에는 항상 "근거 chunk만" 전달
- 자유 추정 문장 비율 제한
- 근거 미검출 시 답변 축소 및 추가질문 유도
- 법률 리스크:
- "법률 자문 대체 아님" 고지
- 최종 의사결정은 인사/법무 승인 절차로 분리
- 개인정보/민감정보:
- 내부DB 비식별화
- 최소 권한 API 키 운영
- 로그 저장 시 PII 마스킹

## 8. 운영 워크플로우
- 실시간 질의:
- 챗 UI -> 정규화 -> 근거 수집 -> 검증 -> 답변
- 정기 동기화:
- `스케줄 설정`으로 법령/판례/내부DB 증분 동기화
- 장애 대응:
- API 실패 시 대체 소스 조회
- 근거 수집 실패 시 "조회 실패 + 재시도 예정" 명시

## 9. 품질 지표 (KPI)
- 근거 충족률: 법령/판례 슬롯 충족 비율
- 무근거 문장률: 최종 출력 중 검열에 걸린 문장 비율
- 내부DB 참조율: 내부 규정/사례가 결합된 응답 비율
- 재생성률: 1차 생성 실패 후 재생성 비율
- 사용자 신뢰지표: 도움이 됨/근거충분 평가 점수

## 10. 단계별 구축 일정 (권장)
- 1단계 (1주): 데이터 소스 연결
- Open API/판례/내부DB/전자소송 안내 연결
- 2단계 (1주): RAG + 템플릿 강제
- 임베딩/근거 결합/출력 슬롯 검증
- 3단계 (1주): 앱 UI + 권한 + 로그
- 애플리케이션 배포, 접근권한, 운영 대시보드
- 4단계 (1주): 시범운영/튜닝
- 환각 케이스 수집, 규칙 개선, 프롬프트 고도화

## 11. 즉시 실행 체크리스트
- 공식 API 키 준비
- 내부DB 최소 스키마 확정
- 챗봇 필수 출력 슬롯 확정
- 근거 검열 규칙(정규식/형식검사) 확정
- 배포 권한정책(누구나/로그인/초대) 확정
- 운영 책임자(법무/인사/개발) 승인 라인 확정

## 12. 결론
- AI Canvas만으로도 징계 챗봇의 핵심 구조(수집-검증-생성-배포)를 구현 가능하다.
- 성공의 핵심은 다음 두 가지다.
- `근거 강제 공급`: 공식 원문 + 내부DB를 시스템이 먼저 제공
- `인용 강제 출력`: 템플릿/검열로 무근거 문장을 차단
